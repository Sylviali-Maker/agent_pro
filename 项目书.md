# NTAD 网络威胁攻防知识图谱问答系统 — 项目书

## 一、项目概述

### 1.1 项目背景

**痛点**：MITRE ATT&CK 是全球最权威的网络安全威胁知识库，涵盖 697 种攻击技术、174 个威胁组织、821 种恶意软件，数据以 STIX 2.0 格式发布，体量达 48.2MB。安全分析师在实际工作中面临三大痛点：① 人工检索效率低，跨实体查询依赖手工拼接；② 关联分析困难，实体间关系链路需逐层翻阅；③ 数据格式非结构化，缺乏统一的查询入口。

**方案**：构建基于知识图谱的智能问答系统，将 STIX 数据导入 Neo4j 图数据库（7 类节点、6 类关系、18,000+ 边），结合 BM25 关键词检索与 LLM 意图识别，实现自然语言提问到结构化回答的端到端链路，支持 7 种意图路由和多路检索融合。

**成果**：系统检索召回率达 96.63%，将原本需要数小时的手工检索缩短至秒级响应，无关实体过滤 90%+，通过 Streamlit 提供 Web 交互界面。

### 1.2 项目目标

构建一个**基于知识图谱的智能问答系统**，支持自然语言提问，自动完成：
- 意图识别（理解用户问什么）
- 多路检索（BM25 关键词 + 图数据库关联查询）
- 智能回答（LLM 整合生成结构化回答）

### 1.3 技术选型

| 组件 | 技术方案 | 选型理由 |
|------|---------|---------|
| 图数据库 | Neo4j 5.x | 原生图存储，Cypher 查询语言，支持多跳关系遍历 |
| 关键词检索 | BM25Okapi (rank-bm25) | 经典词法检索，无需 GPU，中文分词支持好 |
| 中文分词 | jieba | 开源中文分词标准库，支持自定义词典 |
| LLM | 通义千问 qwen-max | 阿里云 API，中文能力强，兼容 OpenAI 接口 |
| Web UI | Streamlit | Python 原生，快速构建聊天界面 |
| 数据源 | MITRE ATT&CK STIX 2.0 | 官方标准格式，结构化程度高 |

### 1.4 工作内容

**知识图谱构建**：解析 48.2MB MITRE ATT&CK STIX 2.0 数据，设计 7 类节点（Technique/Tactic/Group/Malware/Tool/Mitigation/Campaign）和 6 类关系模型，在 Neo4j 中完成 25,842 个对象的导入、唯一约束创建和索引优化，实现多跳关系遍历。

**意图识别模块**：基于通义千问 qwen-max 构建 LLM 意图解析器，通过 6 个 few-shot 示例引导模型输出结构化 JSON（query_type、query_focus、entities、confidence 等 9 个字段），支持 7 种查询意图的自动分类和模糊问题追问，JSON 解析失败时有 fallback 容错机制。

**BM25 检索与 RRF 融合**：对 STIX 数据三遍扫描构建 1,807 篇增强文档（含关联实体名称），实现 jieba 中英文混合分词的 BM25Okapi 检索引擎；引入 Multi-Query 扩展（LLM 生成主查询+1~2 变体）和 Reciprocal Rank Fusion 多路结果融合，k=60 控制排名衰减。

**动态图查询路由**：根据 LLM 输出的 query_focus 字段设计 7 种 Cypher 查询模板（detail/uses/mitigates/belongs_to/attributed_to/tactics_of/general），实现意图驱动的精准图遍历，避免查全量关系，减少无关实体 90% 以上；支持二跳查询自动补充关联实体详情。

**上下文构建与回答生成**：实现 BM25 与图查询结果按 attack_id 去重（图结果优先），设计 6000 字符全局 Token 预算按优先级填充（技术→战术→组织→软件→缓解→活动→关系），最终由 qwen-max 生成 Markdown 结构化回答。

**评估体系搭建**：从 Neo4j 提取 50 条黄金测试集（覆盖 5 种场景），编写评估脚本计算 Precision/Recall/F1 指标，定位并修复 3 个检索链路 Bug（tactics_of 空返回、detail 缺失 Campaign 关系、uses 漏采二跳技术）。

### 1.5 项目成果

**系统能力量化**

| 维度 | 成果 | 量化指标 |
|------|------|---------|
| 知识图谱规模 | 完整覆盖 MITRE ATT&CK Enterprise | 7 类节点 / 6 类关系 / 18,000+ 边 |
| 检索覆盖能力 | BM25 索引覆盖全部实体类型 | 1,807 篇增强文档，含关联实体名称（最多 30 个/文档） |
| 意图理解能力 | 7 种查询意图自动分类 | 含模糊追问和 fallback 容错 |
| 检索质量 | Recall 接近满分 | 整体 Recall = 0.9663，5 种场景中 3 种达 1.00 |
| 查询效率 | 秒级响应 | 从手工跨表检索数小时缩短至即时回答 |
| 上下文质量 | 去重 + 预算控制 | 6000 字符 Token 预算，无关实体减少 90%+ |
| 查询精度 | 动态路由精准过滤 | tactics_of 场景 F1 达 0.93 |

**核心技术创新**

| 创新点 | 说明 | 效果 |
|--------|------|------|
| Multi-Query + RRF 多路检索融合 | LLM 生成主查询 + 变体查询，RRF 合并 | 召回率提升，减少单一表述导致的漏检 |

### 1.6 评估体系

从 Neo4j 提取 50 条黄金测试用例（覆盖 5 种查询场景），对检索链路进行 Precision / Recall / F1 评估，结果如下：

| 指标 | 值 |
|------|-----|
| **Recall（召回率）** | **0.9663** |
| **Precision（精确率）** | 0.3845 |
| **F1 Score** | 0.5501 |

其中最优场景 tactics_of（战术下属技术）F1 达 **0.93**，验证了动态图查询路由的有效性。Precision 偏低主要因 detail / belongs_to 场景图查询返回了大量关联实体（作为 LLM 上下文是正确行为），黄金集仅标记核心实体导致精确率被系统性低估。

## 三、数据模型

### 3.1 数据来源

MITRE ATT&CK Enterprise 数据集，STIX 2.0 格式，包含 **25,842** 个 STIX 对象，涵盖 **15** 种对象类型。

### 3.2 Neo4j 节点模型（7 类）

| 节点标签 | STIX 类型 | ID 前缀 | 数量 | 核心属性 |
|----------|-----------|---------|------|---------|
| **Technique** | attack-pattern | T | 697 | attack_id, name, description, platforms, is_subtechnique |
| **Tactic** | x-mitre-tactic | TA | 15 | attack_id, name, shortname, description |
| **Group** | intrusion-set | G | 174 | attack_id, name, aliases, description |
| **Malware** | malware | S | 726 | attack_id, name, platforms, aliases, description |
| **Tool** | tool | S | 95 | attack_id, name, platforms, aliases, description |
| **Mitigation** | course-of-action | M | 44 | attack_id, name, description |
| **Campaign** | campaign | C | 56 | attack_id, name, aliases, first_seen, last_seen |

### 3.3 Neo4j 关系模型（6 类）

| 关系类型 | 方向 | 数量 | 含义 |
|---------|------|------|------|
| **BELONGS_TO** | Technique → Tactic | 872 | 攻击技术属于某战术阶段 |
| **USES** | Group/Malware/Tool/Campaign → Technique | 18,220 | 使用某攻击技术 |
| **MITIGATES** | Mitigation → Technique | 1,448 | 缓解某攻击技术 |
| **SUBTECHNIQUE_OF** | Technique → Technique | 475 | 子技术关系 |
| **ATTRIBUTED_TO** | Campaign → Group | 26 | 攻击活动归因于某组织 |
| **DETECTS** | DataSource → Technique | (已导入) | 检测关系 |

### 3.4 索引约束

所有 7 类节点均在 `attack_id` 属性上创建唯一约束，保证查询性能和数据一致性。

### 4.2 意图路由表

系统根据 LLM 输出的 `query_focus` 字段，动态选择图查询策略：

| query_focus | 用户意图 | 查询关系 | 跳数 | 示例问题 |
|-------------|---------|---------|------|---------|
| `detail` | 了解实体详情 | 所有关系 | 1-2跳 | "T1059 是什么？" |
| `uses` | 谁在用/用了什么 | USES | 1-2跳 | "APT28 用了哪些技术？" |
| `mitigates` | 防御/缓解 | MITIGATES | 1跳 | "怎么防御 PowerShell？" |
| `belongs_to` | 归属关系 | BELONGS_TO | 1跳 | "T1059 属于哪个战术？" |
| `attributed_to` | 归因溯源 | ATTRIBUTED_TO | 1跳 | "这个活动是谁干的？" |
| `tactics_of` | 战术下的技术 | BELONGS_TO(反向) | 1跳 | "初始访问有哪些技术？" |
| `general` | 通用查询 | 所有关系 | 1-2跳 | "T1059 相关信息" |

### 4.3 Multi-Query 扩展

LLM 意图识别时同时生成 1-2 个变体查询，从不同角度覆盖同义词和相关术语：

```
用户: "PowerShell 攻击怎么防御？"
主查询: ["PowerShell", "defense", "mitigation"]
变体1: ["T1059.001", "constrained", "language mode"]
变体2: ["scripting", "execution", "prevention"]
→ 3 路 BM25 搜索 → RRF 融合
```

### 4.4 RRF 融合算法

Reciprocal Rank Fusion (RRF) 用于合并多路检索结果：

```
rrf_score(document) = Σ 1/(k + rank_i + 1)

k = 60（控制衰减速度）
rank_i = 文档在第 i 路结果中的排名（从 0 开始）
```

同一文档出现在多路结果中时，RRF 分数累加，最终按分数降序排列。

## 五、核心模块详解

### 5.1 意图识别模块 (query_parser.py)

**功能**: 用 LLM 解析用户自然语言提问，提取查询意图和关键实体。

**Prompt 设计要点**:
- 输出严格的 JSON 格式（9 个字段）
- 包含 6 个 few-shot 示例，覆盖所有查询场景
- attack_id 前缀规则：T/TA/G/S/M/C
- 中文提问时自动提取英文关键词
- 模糊问题自动追问（need_clarification）

**容错机制**:
- JSON 解析失败 → 返回 fallback 结果（confidence=0.3）
- LLM 调用失败 → 用原始输入分词作为关键词

### 5.2 BM25 检索模块 (bm25_search.py)

**索引构建**:
- 三遍扫描 STIX 数据：
  1. 构建 stix_id → (attack_id, name) 查找表
  2. 从 relationship 对象提取关联实体名称
  3. 构建文档（含关联名称增强）
- 每篇文档: `{attack_id} {name} {description[:1500]} {platforms} {aliases} {关联名称[:30]}`

**分词策略**:
- 英文: 正则 `[a-z0-9]+` 匹配，转小写
- 中文: jieba 分词
- 过滤单字符 token

**评分归一化**:
- BM25 原始分数 → max-normalization（除以最大值）
- 过滤 normalized_score < 0.01 的极低分

### 5.3 图查询模块 (graph_query.py)

**动态路由**: 根据 `query_focus` 调用不同的 Cypher 查询模板，避免查询无关关系。

**二跳查询**: 一跳发现的关联实体（如组织、软件），自动补充其详情信息。

**实体类型检测**: 根据 attack_id 前缀自动判断实体类型（T=技术, TA=战术, G=组织, S=软件, M=缓解, C=活动）。

### 5.4 上下文构建模块 (answer_generator.py)

**去重逻辑**: BM25 结果与图查询结果按 attack_id 去重，图结果优先（信息更完整）。

**Token 预算**: 全局 6000 字符上限，按优先级填充：
1. 技术详情（15条，描述200字）
2. 战术（10条）
3. 组织（10条，含别名）
4. 软件（15条，标注类型）
5. 缓解措施（10条）
6. 活动（10条）
7. 关系列表（30条，优先级最低）

## 九、部署与运行

### 9.1 环境要求

- Python 3.9+
- Neo4j 5.x（本地或远程）
- 通义千问 API Key

### 9.2 安装步骤

```bash
# 1. 安装依赖
pip install -r requirements.txt

# 2. 配置环境变量
cp .env.example .env
# 编辑 .env 填入 QWEN_API_KEY 和 NEO4J_PASSWORD

# 3. 构建 Neo4j 图数据库
python -m utils.graph_builder

# 4. 构建 BM25 索引
python -m utils.bm25_search build

# 5. 启动 Web UI
streamlit run app.py
```

### 9.3 命令行测试

```bash
# 交互式问答
python -m utils.qa_engine

# Neo4j 查询测试
python -m utils.graph_query --stats
python -m utils.graph_query --attack-id T1059

# BM25 搜索测试
python -m utils.bm25_search search --query "PowerShell" --top-k 5
```
