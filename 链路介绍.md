# NTAD 系统链路介绍

## 整体定位

本系统是一条完整的 **RAG（Retrieval-Augmented Generation）链路**，核心思路是：用 LLM 理解意图，用 BM25 做粗召回，用图数据库做精准关系查询，最后用 LLM 整合生成回答。共 6 个步骤，每步有明确的输入输出，可独立优化。

---

## 链路总览

```
用户提问
   │
   ▼
┌──────────────────────────────────────────────────────┐
│  Step 1 · LLM 意图识别 (qwen-max, temp=0.1)         │
│  输入: 用户自然语言 + 对话历史                         │
│  输出: query_type / query_focus / entities /          │
│        search_keywords / variants / confidence        │
└──────────────┬───────────────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────────────┐
│  Step 2 · BM25 多路检索 + RRF 融合                    │
│  主查询 + 1~2 变体查询 → 每路 top_k=10               │
│  RRF: Σ 1/(60+rank+1) → 去重排序                     │
└──────────────┬───────────────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────────────┐
│  Step 3 · 置信度判断                                   │
│  LLM confidence ≥ 0.5 且 BM25 有结果 → 进入图查询     │
│  否则 → 返回低置信度检索摘要                            │
└──────────────┬───────────────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────────────┐
│  Step 4 · Neo4j 动态图查询 (按 query_focus 路由)      │
│  ┌─────────┬─────────┬──────────┬──────────────┐     │
│  │ detail  │  uses   │mitigates │ belongs_to   │     │
│  │tactics_of│attributed_to│     general        │     │
│  └─────────┴─────────┴──────────┴──────────────┘     │
│  一跳发现关联实体 → 二跳补充详情                        │
└──────────────┬───────────────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────────────┐
│  Step 5 · 上下文构建                                   │
│  BM25 + 图结果按 attack_id 去重 (图优先)              │
│  6000 字符 Token 预算，按优先级填充                     │
│  技术 → 战术 → 组织 → 软件 → 缓解 → 活动 → 关系       │
└──────────────┬───────────────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────────────┐
│  Step 6 · LLM 回答生成 (qwen-max, temp=0.3)          │
│  输入: 用户问题 + 检索上下文 + 对话历史                 │
│  输出: Markdown 结构化回答 (含 attack_id 引用)         │
└──────────────────────────────────────────────────────┘
```

## 分步详解

### Step 1 · LLM 意图识别

**做什么**：理解用户自然语言提问，提取查询意图和关键实体。

**技术方案**：通义千问 qwen-max（兼容 OpenAI 接口），通过 Prompt 工程引导模型输出结构化 JSON。Prompt 中包含 6 个 few-shot 示例，覆盖不同提问场景。

**输出字段（9 个）**：

| 字段 | 含义 | 示例 |
|------|------|------|
| query_type | 问题类型 | "group" / "technique" / "tactic" |
| query_focus | 查询焦点（决定图查询路由） | "uses" / "mitigates" / "detail" |
| entities | 识别出的 MITRE ID | ["G0007"] |
| search_keywords | 主查询关键词 | ["APT28", "attack", "techniques"] |
| search_keywords_variants | 变体查询关键词（1~2 个） | [["G0007", "Fancy Bear"]] |
| confidence | 模型置信度 | 0.95 |
| need_clarification | 是否需要追问 | false |
| clarification_question | 追问内容（如需要） | "请问您想了解哪种攻击技术？" |
| query_type_reasoning | 推理过程 | "用户问的是 APT28 组织..." |

**容错机制**：JSON 解析失败 → 返回 fallback 结果（confidence=0.3）；LLM 调用失败 → 用原始输入分词作为关键词。

---

### Step 2 · BM25 多路检索 + RRF 融合

**做什么**：用关键词在文档库中找到相关实体。

**BM25 检索引擎**：
- 算法：BM25Okapi（经典词法检索，比 TF-IDF 效果更好）
- 索引规模：1,807 篇文档，每篇对应一个 MITRE ATT&CK 实体
- 分词策略：英文正则 `[a-z0-9]+` 转小写，中文 jieba 分词，过滤单字符
- 评分归一化：BM25 原始分数 → max-normalization → 过滤 < 0.01 的极低分

**文档增强**（三遍扫描 STIX 数据）：
1. 构建 `stix_id → (attack_id, name)` 查找表
2. 从 STIX relationship 对象提取关联实体名称（双向）
3. 拼接文档：`{attack_id} {name} {description[:1500]} {platforms} {aliases} {关联名称[:30]}`

这样搜索 "Fancy Bear" 能匹配到 APT28（因为 Fancy Bear 是它的别名，在关联名称中）。

**多路检索**：LLM 在 Step 1 生成了主查询 + 变体查询，每路各搜 top_k=10。

**RRF 融合**（Reciprocal Rank Fusion）：

```
rrf_score(document) = Σ 1/(k + rank_i + 1)

k = 60（控制衰减速度）
rank_i = 文档在第 i 路结果中的排名（从 0 开始）
```

同一文档出现在多路结果中时，RRF 分数累加，最终按分数降序排列。效果：从不同角度覆盖同义词，减少因单一表述导致的漏检。

---

### Step 3 · 置信度判断

**做什么**：判断是否值得走图查询（分流节点）。

**判断逻辑**：
- LLM confidence ≥ 0.5 **且** BM25 有结果 → 进入 Step 4 图查询
- LLM confidence < 0.5 但 BM25 有结果 → 返回低置信度检索摘要
- BM25 无结果 → 返回提示信息

**为什么用 LLM confidence 而非 BM25 分数**：BM25 原始分数不是全局校准的（无关查询可能因 IDF 极端值得到高分），LLM 的置信度更能反映问题的可回答性。

---

### Step 4 · Neo4j 动态图查询

**做什么**：在图数据库中按意图路由，精准遍历关系链路。

**Neo4j 图模型**：

| 节点类型 | 数量 | 示例 |
|---------|------|------|
| Technique（攻击技术） | 697 | T1059 Command and Scripting Interpreter |
| Tactic（战术阶段） | 15 | TA0002 Execution |
| Group（威胁组织） | 174 | G0007 APT28 |
| Malware（恶意软件） | 726 | S0154 Cobalt Strike |
| Tool（工具） | 95 | S0013 PowerSploit |
| Mitigation（缓解措施） | 44 | M1032 Multi-factor Authentication |
| Campaign（攻击活动） | 56 | C0001 等 |

| 关系类型 | 数量 | 方向 |
|---------|------|------|
| BELONGS_TO | 872 | Technique → Tactic |
| USES | 18,220 | Group/Malware/Tool/Campaign → Technique |
| MITIGATES | 1,448 | Mitigation → Technique |
| SUBTECHNIQUE_OF | 475 | Technique → Technique |
| ATTRIBUTED_TO | 26 | Campaign → Group |

**动态路由**（7 种 query_focus）：

| query_focus | 用户意图 | 查询关系 | 示例问题 |
|-------------|---------|---------|---------|
| detail | 了解实体详情 | 所有关系 1-2 跳 | "T1059 是什么？" |
| uses | 谁在用/用了什么 | USES 1-2 跳 | "APT28 用了哪些技术？" |
| mitigates | 防御/缓解 | MITIGATES 1 跳 | "怎么防御 PowerShell？" |
| belongs_to | 归属关系 | BELONGS_TO 1 跳 | "T1059 属于哪个战术？" |
| attributed_to | 归因溯源 | ATTRIBUTED_TO 1 跳 | "这个活动是谁干的？" |
| tactics_of | 战术下的技术 | BELONGS_TO 反向 1 跳 | "初始访问有哪些技术？" |
| general | 通用查询 | 所有关系 1-2 跳 | "T1059 相关信息" |

**二跳查询**：一跳发现的关联实体（如组织、软件），自动补充其详情信息。

**效果**：减少无关实体 90%+，LLM 上下文质量显著提升。

---

### Step 5 · 上下文构建

**做什么**：将 BM25 和图查询结果整合为 LLM 可消费的上下文文本。

**去重逻辑**：收集图查询结果中的所有 attack_id，BM25 结果中已存在的实体直接过滤（图结果优先，信息更完整）。

**Token 预算**：全局 6,000 字符上限，按优先级填充：

| 优先级 | 内容 | 上限 |
|--------|------|------|
| 1 | 技术详情（名称、平台、描述） | 15 条，描述 200 字 |
| 2 | 关联战术 | 10 条 |
| 3 | 关联威胁组织（含别名） | 10 条 |
| 4 | 关联软件（标注类型） | 15 条 |
| 5 | 缓解措施 | 10 条 |
| 6 | 攻击活动 | 10 条 |
| 7 | 关系列表（优先级最低） | 30 条 |

超出预算截断，保证不超 LLM 上下文窗口。

---

### Step 6 · LLM 回答生成

**做什么**：将用户问题 + 检索上下文 + 对话历史打包，生成结构化回答。

**技术方案**：通义千问 qwen-max（temperature=0.3，输出较稳定），Prompt 要求：
- 引用 attack_id（如 T1059、TA0002）
- 使用 Markdown 结构化输出（标题、列表、表格）
- 提供防御建议
- 承认不确定性（检索不充分时明确告知）

**对话历史**：保留最近 6 轮对话作为上下文，支持多轮追问。

**容错**：LLM 调用失败 → 返回检索结果摘要，不丢失已有信息。

---

## 技术栈总结

| 层次 | 技术 | 作用 |
|------|------|------|
| 数据层 | STIX 2.0 JSON (48.2MB) | MITRE ATT&CK 官方数据格式 |
| 存储层 | Neo4j 5.x 图数据库 | 7 类节点、6 类关系、18,000+ 边 |
| 检索层 | BM25Okapi + jieba 中文分词 | 1,807 篇文档的关键词检索 |
| 融合层 | RRF (Reciprocal Rank Fusion) | 多路检索结果合并 |
| 智能层 | 通义千问 qwen-max | 意图识别 + 回答生成 |
| 应用层 | Streamlit | Web 聊天界面 |
